# SDL3 GPU Mandelbrot Demo

Use a Mandelbrot set compute shader for the compute-to-texture
demonstration.  Build a clean standalone sample repository.

## Goal

Build a small C++17 wxWidgets sample program that demonstrates SDL3 GPU
API by:

1. opening a wxWidgets frame,
2. rendering a CPU Mandelbrot image into a plain wx window first,
3. embedding an SDL-backed `wxsdl::SdlCanvas` for GPU presentation,
4. switching between CPU, GPU, height field, and potential field results
   from the menu,
5. creating an SDL GPU device,
6. dispatching a compute shader,
7. writing smoothed Mandelbrot iteration and potential values into GPU
   textures,
8. displaying those textures through simple graphics pipelines,
9. controlling the Mandelbrot view and height camera with wx mouse
   events.

The code should remain small enough to explain in one meeting.

## Current progress

The repository now has the foundation targets needed for the demo:

* `sdlcpp` owns small RAII wrappers around SDL handles and SDL GPU
  handles.
* `core` owns the Mandelbrot parameter block, CPU pixel evaluator,
  iteration buffer, and CPU color-mapped image buffer.
* `wxsdl` owns `wxsdl::SdlCanvas`, a `wxWindow` that attaches an SDL
  window to the native wx widget on Win32, Cocoa, and GTK/X11.
* `mandel` is a wxWidgets GUI executable.  Its main frame creates a
  plain CPU display `wxWindow` and an `SdlCanvas`.
* `mandel` owns a small render host that draws CPU Mandelbrot output into
  the CPU display window, keeps CPU presentation selected by default, and
  switches between CPU, GPU, height field, and potential field
  presentation from the View menu.  Mouse drag pans the 2D views, the
  wheel zooms the 2D views, drag orbits the height views, and the wheel
  changes the height camera distance.  The GPU display path owns an SDL
  GPU device, claims the SDL canvas window, loads static shader assets
  generated by shadercross, creates a Mandelbrot compute pipeline, a
  fullscreen triangle pipeline, and a height field pipeline, writes
  smoothed iteration values and potential values into `R32_FLOAT`
  textures, and presents those textures from the GPU paint handler.
* Unit tests cover the move-only wrapper shape, `SdlCanvas` inheritance
  contract, CPU Mandelbrot behavior, CPU iteration buffers, CPU color
  mapping, viewport pan/zoom math, and the sdlcpp wrapper method surface.

Future Mandelbrot code should build on this structure.  Do not go back to
a raw SDL-only application unless the wx host is deliberately removed.

## Existing support targets

The current low-level targets are:

```cmake
add_subdirectory(core)
add_subdirectory(sdlcpp)
add_subdirectory(wxsdl)
```

`sdlcpp` is the SDL C API wrapper layer.  It already provides wrappers for
the SDL library lifetime, properties, windows, GPU devices, command
buffers, textures, buffers, shaders, samplers, pipelines, passes, fences,
transfer buffers, swapchain textures, and GPU window claims.

`core` is the domain layer.  Keep Mandelbrot parameters, viewport mapping,
iteration/color data, CPU reference code, and render input/output types
there.

`wxsdl` is the wx/SDL bridge.  Its public surface is currently
`wxsdl::SdlCanvas`, which exposes the attached `sdlcpp::Window`.

## Mandel parameter block

Use a simple parameter block for both the CPU reference code and the GPU
shader.

C++ version:

```cpp
#pragma once

namespace core
{

struct MandelParams
{
    double x_min;
    double y_min;
    double dx;
    double dy;
    double z0_real;
    double z0_imag;
    double bailout;
    double close_enough;
    int width;
    int height;
    int max_iterations;
    int first_saved_and;
    int periodicity_next_saved_incr;
    int periodicity_check;
};

} // namespace core
```

Initial values:

```cpp
MandelParams params{
    -2.5,     // x_min
    -1.5,     // y_min
    4.0 / 800.0,
    3.0 / 600.0,
    0.0,      // z0_real
    0.0,      // z0_imag
    4.0,      // bailout
    1.0e-7,   // close_enough
    800,
    600,
    1000,
    15,       // first_saved_and
    10,       // periodicity_next_saved_incr
    1         // periodicity_check
};
```

## CPU API

`MandelCpu.h`:

```cpp
#pragma once

#include <core/MandelIterationBuffer.h>
#include <core/MandelParams.h>

namespace core
{

float mandel_cpu_pixel(const MandelParams &params, int px, int py);
MandelIterationBuffer render_mandel_cpu(const MandelParams &params);

} // namespace core
```

`MandelCpu.cpp` implements a straightforward Mandelbrot pixel
evaluator and renders it into a CPU iteration buffer:

1. map pixel coordinate to complex coordinate `c`,
2. initialize `z = c + z0`,
3. iterate `z = z*z + c`,
4. stop when `x*x + y*y >= bailout`,
5. optionally stop early when periodicity is detected,
6. return a smoothed float iteration value or `max_iterations` for inside
   points.

The GPU shader should match this CPU reference implementation.  Avoid
unrelated engine state.

## CPU periodicity checking

Use per-pixel periodicity checking in the sample.  Do not add cross-pixel
adaptive state.

Reference pseudocode:

```text
saved_x = 0
saved_y = 0
saved_and = first_saved_and
saved_incr = 1

for iter in 1 .. max_iterations:
    z = z*z + c

    if norm(z) >= bailout:
        return continuous_iteration(iter, norm)

    if periodicity_check:
        if (iter & saved_and) == 0:
            saved_x = z.x
            saved_y = z.y
            saved_incr = saved_incr - 1

            if saved_incr == 0:
                saved_and = (saved_and << 1) + 1
                saved_incr = periodicity_next_saved_incr
        else:
            if abs(saved_x - z.x) < close_enough and
               abs(saved_y - z.y) < close_enough:
                return max_iterations

return max_iterations
```

This keeps the algorithm simple and GPU-friendly.

## Tool target

`tools/CMakeLists.txt`:

```text
configure:
    copy chroma.map into the build assets directory
    configure assets.h with that runtime assets directory

build:
    shadercross compiles fixed HLSL files to DXIL, SPIR-V, and MSL
    mandel depends on the generated shader assets

runtime:
    SDL chooses the GPU backend
    mandel loads the matching static shader asset
    mandel creates shaders through sdlcpp GPU device methods
```

The current tool is the wx application shell and render host with CPU,
GPU, height field, and potential field presentation modes.  The GPU path
dispatches a fixed HLSL compute shader, writes continuous iteration and
potential textures, displays the iteration texture with a fullscreen
triangle, and displays height modes with a generated grid.  Keep SDL
window access through `wxsdl::SdlCanvas::window()`.

## Presentation model

The frame should contain two separate display windows:

```text
plain wxWindow       CPU result
wxsdl::SdlCanvas     GPU result
```

The CPU result is selected by default.  Render CPU iteration values into
a buffer, map those values to colors in a `wxImage` or `wxBitmap`, and
copy that image in the CPU display window's paint handler.

The View menu switches between CPU, GPU, height field, and potential
field results.  Keep both display windows in the same frame and use
visibility to choose the active presentation:

```text
CPU selected:           CPU wxWindow shown, GPU SdlCanvas hidden
GPU selected:           CPU wxWindow hidden, GPU SdlCanvas shown
Height Field selected:  CPU wxWindow hidden, GPU SdlCanvas shown
Potential Field selected:  CPU wxWindow hidden, GPU SdlCanvas shown
```

Do not mix CPU drawing into the SDL canvas, and do not use the GPU
swapchain as the CPU display path.  `Height Field` is a View menu item
that reuses the current GPU iteration texture and presents it as a
procedural mesh.  `Potential Field` uses the same mesh presentation with
a separate GPU potential texture as the height source.

## CPU frame flow

The initial visible path is CPU-rendered.  The wx event loop drives the
CPU display window.

```text
wx dispatches paint, size, timer, and mouse events

if size or parameters changed:
    render MandelCpu output into a CPU iteration buffer
    map iteration values to colors in a wxImage or wxBitmap

paint CPU display window:
    draw the wxImage or wxBitmap
```

Handle resize from wx size events.  Handle mouse input through wx mouse
events on the active display window.  Do not add a raw SDL event loop.

## SDL3 GPU frame flow

The runtime frame flow should be driven by the wx event loop.  The main
frame owns the canvas and a small render host attached to the display
windows.

```text
wx dispatches paint, size, timer, and mouse events

if parameters changed:
    push MandelParams as compute uniform data

command_buffer = device.AcquireGPUCommandBuffer()

begin compute pass
    bind Mandelbrot compute pipeline
    bind iteration and potential textures as writable storage textures
    dispatch ceil(width / 16), ceil(height / 16), 1
end compute pass

acquire swapchain texture

begin render pass
    bind fullscreen blit pipeline
    bind iteration texture
    draw fullscreen triangle
end render pass

submit command buffer
```

The GPU presentation mode blits the iteration texture through a fullscreen
triangle.  The height field and potential field modes render a generated
triangle grid using the iteration or potential texture as the height
source and the iteration texture for palette color.

Handle resize from wx size events.  Handle mouse input through wx mouse
events on the active display window.  Do not add a second raw SDL event
loop.

## Input ownership

Keyboard and mouse input are handled by wxWidgets.  SDL is used for the
window handle and GPU presentation path, not for application input.  The
Mandel app disables SDL keyboard, text, and mouse events.  Do not poll or
process SDL input events in the Mandelbrot tool.

## Texture strategy

Start with a two-stage output path on both CPU and GPU:

1. Mandelbrot evaluation writes smoothed float iteration values,
2. a separate presentation step maps iteration values to color.

This keeps the fractal math separate from palette display.

Possible future texture formats:

```text
R32_UINT       raw iteration count
R32_FLOAT      continuous iteration count
RGBA8         direct color output
RGBA16_FLOAT  HDR-ish display path
RGBA32_FLOAT  full float accumulation experiments
```

Use `R32_FLOAT` for the current continuous iteration output and potential
height output.

## Compute shader sketch

`mandel.comp.hlsl`:

```hlsl
struct MandelParams
{
    float4 view;
    float4 constants;
    float4 potential;
    int4 image;
    int4 periodicity;
};

RWTexture2D<float> iteration_tex : register(u0, space1);
RWTexture2D<float> potential_tex : register(u1, space1);
ConstantBuffer<MandelParams> params : register(b0, space2);

float continuous_iteration(int iter, float norm);
float potential_height(int iter, float norm);

[numthreads(16, 16, 1)]
void main(uint3 tid : SV_DispatchThreadID)
{
    if (tid.x >= uint(params.image.x) || tid.y >= uint(params.image.y))
    {
        return;
    }

    float cx = params.view.x + float(tid.x) * params.view.z;
    float cy = params.view.y + float(tid.y) * params.view.w;

    float x = cx + params.constants.x;
    float y = cy + params.constants.y;

    float saved_x = 0.0;
    float saved_y = 0.0;
    int saved_and = params.image.w;
    int saved_incr = 1;
    bool iteration_written = false;

    for (int iter = 1; iter < params.image.z; ++iter)
    {
        float x2 = x * x;
        float y2 = y * y;
        float xy = x * y;

        x = x2 - y2 + cx;
        y = 2.0 * xy + cy;

        float norm = x * x + y * y;
        if (!iteration_written && norm >= params.constants.z)
        {
            iteration_tex[tid.xy] = continuous_iteration(iter, norm);
            iteration_written = true;
        }

        if (norm >= params.potential.x)
        {
            if (!iteration_written)
            {
                iteration_tex[tid.xy] = continuous_iteration(iter, norm);
            }
            potential_tex[tid.xy] = potential_height(iter, norm);
            return;
        }

        if (!iteration_written && params.periodicity.y != 0)
        {
            if ((iter & saved_and) == 0)
            {
                saved_x = x;
                saved_y = y;
                --saved_incr;

                if (saved_incr == 0)
                {
                    saved_and = (saved_and << 1) + 1;
                    saved_incr = params.periodicity.x;
                }
            }
            else if (abs(saved_x - x) < params.constants.w &&
                     abs(saved_y - y) < params.constants.w)
            {
                iteration_tex[tid.xy] = float(params.image.z);
                potential_tex[tid.xy] = 0.0;
                return;
            }
        }
    }

    if (!iteration_written)
    {
        iteration_tex[tid.xy] = float(params.image.z);
    }
    potential_tex[tid.xy] = 0.0;
}
```

## Blit shaders

Use a fullscreen triangle.  The vertex shader can synthesize positions
from `SV_VertexID`; no vertex buffer is needed.

The fragment shader should sample or load the iteration texture and
convert iteration values to color with the same `chroma.map` palette used
by the CPU path.

Coloring:

```text
iteration >= max_iterations -> black
otherwise                   -> interpolate chroma.map by fractional
                               palette position
```

The height vertex shader builds a grid from `SV_VertexID`, samples either
the continuous iteration texture or the potential texture as height, and
uses a depth target while the fragment shader reuses the iteration texture
and palette buffer for color.

## Runtime controls

Minimal controls:

```text
2D CPU/GPU drag       pan
2D CPU/GPU wheel      zoom in/out
height drag           orbit camera
height wheel          move camera closer/farther
```

Handle these through wx mouse events on the active display window.  Do
not implement a full UI.  This is a graphics API sample, not a full
fractal application UI.
